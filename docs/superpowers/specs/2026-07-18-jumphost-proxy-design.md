# 跳板机 + 代理设置 设计文档

日期：2026-07-18
状态：待确认

## 背景

有些服务器在内网，本机无法直接连上，需要借助一台跳板机（堡垒机）中转，或者通过 SOCKS5 / HTTP 代理才能访问。编辑连接弹窗中已预留"跳板机"和"代理设置"两个占位 tab，本次实现它们。

## 目标

- 连接内网服务器时，可以通过另一台已保存的服务器做跳板中转
- 也可以通过 SOCKS5 / HTTP 代理建立连接
- 配置方式与现有表单风格一致，测试连接、测延迟、终端连接全部生效

## 需求范围

### 做

1. 连接配置新增跳板机设置：从已保存的连接中选择一个作为跳板（先连跳板，再通过跳板连目标机）
2. 连接配置新增代理设置：类型（SOCKS5 / HTTP）、地址、端口、可选的用户名密码
3. 跳板机和代理互斥：一个连接同一时间只能启用其中一种（界面上用单选切换）
4. 编辑弹窗启用"跳板机"和"代理设置"两个 tab
5. 测试连接、列表测延迟、打开终端，全部按配置走跳板/代理
6. 删除某个连接时，自动清除其它连接对它的跳板引用

### 不做

- 多级跳板串联（只支持一级跳板）
- 跳板机和代理叠加使用
- 自动代理（PAC）、全局代理（代理按连接单独配置）
- 跳板机的独立密码记忆以外的凭据同步（跳板机就是一条普通连接，凭据用它自己的）

## 技术设计

### 数据结构

`SSHConnection` 新增字段：

```ts
// 二选一，互斥
jumpHostId?: string | null   // 作为跳板的另一个连接的 id
proxy?: {
  type: 'socks5' | 'http'
  host: string
  port: number
  username?: string
  password?: string
} | null
```

### 连接建立流程（主进程 sshManager）

1. 无跳板无代理：保持现状，直连
2. 有跳板：
   - 先按跳板连接的配置建立 SSH 连接（跳板自身若也有跳板/代理配置，忽略之，只支持一级）
   - 通过跳板连接的 `forwardOut(目标host, 目标port)` 拿到数据通道
   - 把该通道作为 `sock` 传给目标机的连接配置，建立目标连接
   - 目标连接断开时，跳板连接一并关闭
3. 有代理：
   - SOCKS5：引入 `socks` 依赖包，用 `SocksClient.createConnection` 拿到通道作为 `sock`
   - HTTP：用 `net` 模块向代理发 `CONNECT host:port` 建立隧道，拿到通道作为 `sock`
   - 之后 SSH 连接照常建立在通道之上

### 界面

编辑弹窗：

- "跳板机"tab：启用开关 + 下拉框（列出除当前连接外的所有已保存连接，显示名称和地址）
- "代理设置"tab：启用开关 + 类型下拉（SOCKS5 / HTTP）+ 地址、端口输入框 + 用户名、密码（可选）
- 两个 tab 的开关互斥：打开一个时另一个自动关闭并给出提示
- 列表和终端页不展示跳板/代理信息（保持简洁），编辑弹窗中可查看

### 错误处理

| 场景 | 处理 |
| --- | --- |
| 跳板连接失败 | 提示"跳板机连接失败：原因"，不再尝试目标机 |
| 目标机连接失败 | 提示目标机错误原因，跳板连接随之关闭 |
| 代理连接失败/认证失败 | 提示"代理连接失败：原因" |
| 跳板下拉中选择了已被删除的连接 | 保存时校验，提示并清空该字段 |
| 删除被引用的连接 | 删除时把所有 `jumpHostId` 指向它的连接该字段置空 |

### 测试与验证

1. `npx tsc -b` 类型检查通过
2. 浏览器 mock 模式验证表单交互（tab 启用、互斥、保存回填）
3. 真实环境验证（需要两台服务器或一台可当中转的机器）：
   - 配置跳板后测试连接成功，终端可正常交互
   - 配置 SOCKS5 代理后测试连接成功
   - 跳板/代理配置错误时，错误提示符合上表
   - 删除跳板机后，引用它的连接跳板字段被清空

## 依赖变化

- 新增 npm 依赖：`socks`（SOCKS5 代理通道）
